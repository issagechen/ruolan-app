## 1. 项目概述

### 1.1 产品定位
「若澜」是一款**运行在安卓平板上的本地化 AI 智能伴侣应用**。核心卖点是：**无需联网、隐私安全、可语音实时对话、可自定义形象与性格的虚拟陪伴角色**。

### 1.2 目标用户
- 需要情感陪伴 / 轻量咨询的个人用户
- 对隐私敏感、不愿将对话上传云端的用户
- 拥有中端以上安卓平板（ARM64）的硬件环境

### 1.3 技术栈与运行环境
| 项 | 现状 |
|---|---|
| 框架 | Flutter 3.x（Dart SDK ≥3.2） |
| 本地推理 | llama.cpp（C++ JNI，`libllama_wrapper.so`），模型 DeepSeek-R1-Distill-Qwen-7B-Q4_K_M.gguf |
| 语音输出 | `flutter_tts`（系统 TTS） |
| 语音输入 | Android 原生 `SpeechRecognizer`（MethodChannel，未用 `speech_to_text` 包） |
| 本地存储 | `sqflite`（聊天记录） + `shared_preferences`（角色配置） |
| 远程兜底 | OpenAI 兼容 `/chat/completions` 流式接口（`ApiConfig.mode='local'` 默认关闭） |
| 形态 | 横屏平板布局（左对话面板 + 右形象面板），`singleTop` Activity |

### 1.4 关键约束
- **仅支持 Android（arm64）**：iOS / Windows / Web 代码存在但未接入原生推理，属于占位。
- **模型文件大（约 4–5GB）**：需外部拷贝到 `/sdcard/models/` 或 `filesDir/models/`，首次启动自动复制。
- **性能依赖硬件**：7B 模型在平板上 token 速度有限，`n_threads=4`、`ctx=1024` 为当前保守配置。

## 3. 问题诊断（技术债 / 缺陷 / 风险）

| 编号 | 类型 | 问题 | 影响 | 严重度 |
|---|---|---|---|---|
| D-01 | 缺陷 | DeepSeek-R1 推理会输出 `<think>…</think>` 思考块，当前**原样透传并展示/存储**给用户 | 用户体验割裂、泄露内部推理 | 高 |
| D-02 | 配置 | `ApiConfig.localCtxSize=1024` 过小；`maxTokens` 在 `llm_service`(512) 与 `chat_service`(512) 与历史补丁(1024) 不一致 | 长对话极易被截断 | 中 |
| D-03 | 健壮 | 模型加载失败仅以 `【加载失败】` 文本混入对话流，无重试/引导 UI | 首启失败即“假死” | 高 |
| D-04 | 边界 | 上下文溢出仅简单裁剪最早消息，未分段、未摘要；`buildChatPrompt` 不含 system 之外的历史管理 | 长会话记忆断裂 | 中 |
| D-05 | 冗余 | `pubspec.yaml` 声明 `speech_to_text` 但 STT 走原生 Channel，依赖**未使用** | 包体积/维护噪音 | 低 |
| D-06 | 架构 | 包名歧义：`com.ruolan.app`（MainActivity）与 `com.ruolan.ruolan_app`（Bridge）并存 | 维护易错 | 低 |
| D-07 | 体验 | `AgentProfile.suggestedReplies` 与 `openingLine` 已存但**未在任何 UI 渲染** | 设定功能“半成品” | 中 |
| D-08 | 体验 | 形象面板为静态图，无动画/表情/状态联动 | 陪伴感弱 | 低 |
| D-09 | 边界 | 语音识别错误/无结果时无提示，直接 `continue` 重试 | 静默失败 | 中 |
| D-10 | 可维护 | 无自动化测试、无 CI、无版本化需求文档（本文件即补此缺口） | 回归风险 | 中 |

## 6. 详细需求规格

> 字段：需求ID / 模块 / 描述 / 优先级 / 验收标准 / 依赖

### 6.1 本地推理（INF）
- **REQ-INF-001** 模型加载与卸载 | P0 | 启动后可加载指定 GGUF 并卸载；验收：调用 `loadModel` 返回 true，推理可出 token，`unload` 后资源释放无泄漏。| 依赖：原生 `libllama_wrapper`
- **REQ-INF-002** 流式推理 | P0 | 逐 token 回调并经 EventChannel 上送 Dart。验收：UI 实时显示增量文本，无整段延迟。| 依赖：REQ-INF-001
- **REQ-INF-003** 上下文窗口与配额 | P1 | 提供可配置 `ctxSize`/`maxTokens`，默认提升（建议 ctx≥2048、maxTokens≥1024）。验收：长对话不被过早截断；参数可从设置页调整并生效。| 依赖：REQ-SET-003
- **REQ-INF-004** 思考过程过滤 | **P0** | 对 DeepSeek-R1 类模型的 `<think>…</think>` 在**流式展示与存储时隐藏**，不进入用户可见文本与历史。验收：界面与历史库均不含 `<think>` 块；推理结束后正确衔接后续正文。| 依赖：REQ-INF-002
- **REQ-INF-005** 加载失败可恢复 | **P0** | 模型缺失/加载失败时给出明确引导（路径说明/重试按钮），**不**污染对话流。验收：失败时不出现 `【加载失败】` 混入气泡；提供“重试/选择模型”入口。| 依赖：REQ-INF-001、REQ-INF-006
- **REQ-INF-006** 模型管理 | P1 | 列出可用模型、切换、显示大小/量化信息；缺失时引导拷贝。验收：可在设置内切换模型并热重载。| 依赖：REQ-INF-001

- **REQ-CHAT-004** 上下文裁剪策略 | P1 | 长会话按轮次裁剪并可选摘要。验收：超出 ctx 时最近 N 轮完整、更早摘要或丢弃，且不影响连贯性。| 依赖：REQ-INF-003
- **REQ-CHAT-005** 开场白展示 | **P1** | 新会话首位展示角色 `openingLine`。验收：清空/新会话后首条为开场白气泡。| 依赖：REQ-SET-001

## 7. 边界场景与异常处理

| 场景 | 预期处理 | 关联需求 |
|---|---|---|
| 模型文件缺失/损坏 | 引导拷贝 + 重试入口，不污染对话 | REQ-INF-005、REQ-INF-006 |
| 模型加载中用户发消息 | 排队或提示“加载中”，加载完再答 | REQ-INF-005 |
| 推理过程被 `stop` | 立即停止、补全当前句、标记中断 | REQ-CHAT-008、REQ-INF-002 |
| 上下文超出 ctx | 按 REQ-CHAT-004 策略裁剪/摘要 | REQ-CHAT-004 |
| 网络（远程）中断 | 提示“网络错误”并可重发 | REQ-API-001 |
| 语音识别无结果 | 明确提示，限次重试后退出本轮 | REQ-VOC-004 |
| 麦克风未授权 | 权限引导，拒绝后禁用通话并说明 | REQ-VOC-003 |
| TTS 初始化失败 | 静默降级，不阻断对话 | REQ-VOC-001 |
| 配置为空（首装） | 使用 `PersonaConfig.defaultPersona` 默认值 | REQ-AVT-001 |
| 数据库升级 | `onUpgrade` 迁移，失败回退并清库提示 | REQ-STO-002 |

## 8. 需求依赖关系

```
原生推理层 (libllama_wrapper / LlamaBridge)
   └─ REQ-INF-001/002/003
        ├─ REQ-INF-004 (思考过滤，依赖流式)
        ├─ REQ-INF-005 (失败处理，依赖加载)
        └─ REQ-INF-006 (模型管理，依赖加载)
存储层 (sqflite / shared_preferences)
   └─ REQ-STO-001
        ├─ REQ-CHAT-002/004/007
        ├─ REQ-STO-002 (迁移)
        └─ REQ-AVT-001 (配置持久化)
对话层
   ├─ REQ-CHAT-001/003 (P0, 已具备)
   ├─ REQ-CHAT-005/006 (依赖 REQ-SET-001 角色配置)
   └─ REQ-CHAT-008/009 (P2)
语音层
   ├─ REQ-VOC-001/002 (P0)
   └─ REQ-VOC-003/004 (依赖 001/002)
设置层
   ├─ REQ-SET-001 (P0) ──支撑 REQ-CHAT-005/006、REQ-AVT-001
   ├─ REQ-SET-003 (推理参数) ──前置 REQ-INF-003
   └─ REQ-SET-002 (诊断)
远程兜底
   └─ REQ-API-001/002 (依赖本地形态稳定后)
```

**依赖合理性校验：**
- ✅ 原生层先于一切 Dart 功能 —— 合理。
- ✅ `REQ-SET-001` 先于 `REQ-CHAT-005/006` —— 合理（建议回复数据来自设置）。
- ✅ `REQ-INF-003` 先于 `REQ-SET-003` 生效 —— 合理，但存在**轻微倒置**：参数 UI 可先建，生效依赖原生读取；建议将“参数读取”作为原生小改造一并排入 P1。
- ⚠️ `REQ-CHAT-007`（多会话）依赖 `REQ-STO-001` 当前单表结构，需在 `messages` 表增加 `session_id` 列 —— 已通过 REQ-STO-002 迁移覆盖，依赖闭环成立。
